# Self-Optimization Patterns

Workflow patterns that improve agent efficiency. Reviewed at session start, updated at session end.
Patterns promoted from "observed" to "validated" after 3+ confirmations.
Max 15 patterns — unvalidated patterns older than 30 days are dropped.

## How to Use

1. **Session start:** Scan validated patterns for anything relevant to the current task
2. **During work:** Apply relevant patterns proactively
3. **Session end:** Add new patterns as "observed", increment count on existing ones, promote at 3+

| **Census by grepping the NODE NAME and reading every hit — never by a regex that requires the write verb and the path in one expression.** The PR-1 census used `(set\|update\|push\|remove)\(\s*ref\((rtdb\|database),\s*[^)]*<NODE>` and returned **zero hits** for `whatsapp-numbers`, directly contradicting an audit that named live writers by line number. The audit was right: the real code splits the statements — `const numbersRef = ref(rtdb,'whatsapp-numbers')` on one line, `await push(numbersRef, data)` in a *different function* 500 lines later. **A zero-hit census is the most dangerous possible result**, because it reads as "no consumers, safe to lock" when it actually means "my regex doesn't match this codebase's idiom." Corollaries: (a) when a census contradicts an audit/report, assume the CENSUS is wrong until proven otherwise — the audit was written by someone reading code, the regex by someone predicting how it would be written; (b) the node name is the one token that must appear literally, so it is the only reliable anchor; (c) still `--include=*.js --include=*.html --include=*.vue` per the #125 refinement.
