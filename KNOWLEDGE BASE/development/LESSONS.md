| 2026-06-02 | tools/rtdb-write-windows | **To write a single RTDB value from a Windows dev box, a direct REST `PATCH` with a `gcloud auth print-access-token` token is the reliable method — the other paths all fail.** PR5 deploy (setting `subscriptionTiers/free/limits/maxWorkflows = 5`): (a) the firebase MCP `realtimedatabase_set_data`/`get_data` tools errored internally (`Cannot read properties of undefined (reading 'join')`); (b) `firebase database:update <path>` rejects stdin on Windows (`STDIN input is not available`); (c) per the 2026-06-01 lesson, PowerShell `Out-File -utf8` adds a BOM that breaks firebase's JSON parse, and git-bash MSYS mangles the `/path`. | 2026-06-02 | rtdb/rules-validate | **A parent-level `.validate` using `newData.hasChild('X')` to reject field X is BUGGY for multi-path `update()` — `newData` at the parent reflects the POST-MERGE state, so it "sees" X whenever X already exists in the DB, even when the write doesn't touch X.** PR4 entitlement lock: the merged plan's §2 proposed `subscriptions/$uid .validate: "... && !newData.hasChild('features') && !newData.hasChild('limits')"` to make the resolver the sole writer. That would have false-rejected EVERY admin status/extend write on any already-materialized subscription. Correct tool: **child-level `"features": { ".validate": false }`** — it only evaluates when the write tree actually includes data at/under that node, so admin writes to other fields pass while any client write to features/limits is rejected. And child `.validate:false` IS effective (unlike child `.write:false`, the dead-rule trap) — `.validate` does not cascade-grant. Caught at ground-truth/implementation time, not in the merged plan. **Heuristic: to forbid a client from writing a specific subtree, use child `.validate:false` on that subtree, NOT a parent `!newData.hasChild()` guard.** | Session: PR4 rule lock (#125) |

| 2026-05-31 | rtdb/rules-cascade | **RTDB read permission CASCADES from a parent, and a query is authorised at the QUERIED node — rules cannot filter query results.** Two consequences that drove the whole H-2 design (PR #96): (1) a root `".read": "auth != null"` makes every properly-scoped per-child rule below it DEAD — the node is fully readable regardless.
