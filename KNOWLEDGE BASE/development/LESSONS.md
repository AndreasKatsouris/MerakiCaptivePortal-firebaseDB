| 2026-05-15 | process/pre-merge-audit | **A mid-PR reflect commit ≠ a code-review fold-in. Before merge, re-audit the automated review against the current branch state — even if the operator says "looks good".** PR #65 had 7 automated-review findings (1 🔴 must-fix + 2 🟡 + 4 🔵). The mid-PR reflect commit body documented the *preview-smoke* fix-ups (server-shape mismatches) but addressed ZERO of the review items. The must-fix (onSave drops `commitResponse` return → server-422 safety net silently dead) would have shipped to master if operator hadn't asked "have all the code-reviewed issues been resolved?" before merging. **Heuristic at PR-ready-to-merge time:** `gh pr view N --json reviews --jq '.reviews[0].body'` → for each finding, grep current branch state → answer YES/NO/PARTIAL with file:line evidence — even when the PR has subsequent commits that look like fixes. The commit messages of post-review commits described preview fix-ups, not review-item fix-ups; that mismatch is the tell. Sister to 2026-05-11 "structural-vs-content" (presence of fix commits ≠ presence of fixes for review items). | Session: post-PR #58 reflect → pre-merge PR #65 audit |
| 2026-05-15 | tools/git-rebase | **Squash-merged sibling reflect commits cause predictable doc conflicts on rebase — they always touch the same 4 feedback files + project-status.json.** PR #65's mid-PR reflect commit (876cf206) conflicted with master's PR #64 housekeeping reflect on exactly these 5 files: PROJECT_BACKLOG / LESSONS / SCORECARD / SELF_OPTIMIZATION / project-status.json. Both sides ADD entries at the top of append-style tables (LESSONS / SCORECARD / SELF_OPT) → resolution is mechanical: strip conflict markers, keep both blocks chronologically. PROJECT_BACKLOG "Last updated" and project-status.json "lastUpdatedNote" are single-string fields → write one combined line capturing both states. **Heuristic for the rebase:** for append-style tables use `awk '/^<<<<<<< HEAD$/ {next} /^=======$/ {next} /^>>>>>>> / {next} {print}'` to drop markers and keep both sides; manually merge the single-field conflicts. Avoids the 30-minute conflict-resolution trap when N reflect cycles accumulate against an inflight branch. | Session: PR #65 rebase against post-PR #64 master |
| 2026-05-13 | process/worktree-audit | **Old `.worktrees/*` folders can hide legitimately unmerged commits — a worktree's branch may have N commits, of which only some are squash-equivalent on master.** Audited 6 leftover worktrees this session; 4 were fully merged (tree match anywhere on master), but 2 had real unmerged content: `docs/post-pr58-reflect` had a `project-status.json` sync that fell out of PR #59 (the KB feedback files made it but the JSON didn't); `feature/ross-v2-actions` had a UX fix (snooze + nav hrefs on Ross v2 fallback cards) committed 1 hour AFTER PR #20 merged that was never opened as its own PR. Cherry-picked both as PRs #60/#61. **Heuristic before deleting a worktree:** `git cherry origin/master <branch>` lists per-commit patch-equivalence (lines with `+` are NOT on master) — but it's MISLEADING for squash-merged branches because N small patches don't pattern-match a single squash. Combine with `git log origin/master --format='%T' \| grep <branch_tree>` (tree match anywhere on master = squashed-and-merged). Use BOTH heuristics; trust tree-match over cherry. | Session: housekeeping 2026-05-13 |
| 2026-05-13 | process/session-opening | **`git fetch` should fire BEFORE reading `PROJECT_BACKLOG.md` at session start — otherwise the "what" is one merge behind reality.** This session: I read backlog and stated sprint goal "day-zero auto-activation is next" — but day-zero had been merged (PR #58) hours earlier and PR #59 had updated the backlog accordingly. I was working from a stale 1-commit-behind copy until the operator asked "are we up to date with the repo?" The literal Session Opening Protocol mandates reading BACKLOG but doesn't say `git fetch` first. Heuristic: step 0 should be `git fetch origin && git status` so a behind-master local check happens BEFORE reading any of the four feedback files. | 2026-05-13 | process/test-mocks | **Test mocks shaped from documentation lie — copy the server's actual `res.json(...)` line verbatim into the mock.** PR #65 hit THREE consecutive server-shape mismatches across one PR, all caught only by operator preview (not by 36 passing unit tests, not by spec review, not by code review): (1) `getPlaybookWorkflows` returns `tasks: loc.tasks \|\| {}` — RTDB-shaped object keyed by taskId, not an array; my mocks used `tasks: [{...}]`. (2) `rossSubmitResponse` returns `{ success, taskId, flagged, runCompleted }` — NOT the full `responses` map; my code expected `result.responses[taskId]` and `result.status === 'completed'`. (3) `rossCreateRun` returns `{ success, runId, run, created }` — the actual run is nested under `.run`; my code assumed it was flat. Each time tests passed because mocks were shaped to my assumption rather than the server's actual response. **Heuristic: when writing a vitest mock for a CF wrapper, OPEN `functions/ross.js`, find the `res.json({ result: ... })` line, and copy the object shape verbatim into the mock's `Promise.resolve({...})`. If the code path is conditional (e.g. resume vs create), copy BOTH lines.** Sister to the 2026-05-01 "verify field names against server seed" pattern — same discipline applied to RUNTIME response shape, not just stored field names. Cost of skipping: 3 round-trips of preview testing + ~8 fix-up commits + operator's afternoon. | Session: PR #65 (operator preview ×3) |
| 2026-05-13 | tools/git | **The `block-master-commit` hook resolves `cd <path>` from the shell's starting dir; `cd C:/...worktrees/...` from a master-shell triggers it.** When the harness opens a Bash subprocess in the master worktree and you run `cd .worktrees/<name> && git commit ...`, the hook sees a master shell + a destructive git op and blocks even though the resolved cwd is a feature branch. Workaround: `git -C <full-path> commit ...` bypasses the cd entirely and the hook reads the explicit path. The same applies to `git -C <path> add`, `git -C <path> push`, etc. Worth promoting into the worktree workflow as the default git invocation form when operating across worktrees from a single Claude Code session. | Session: PR #65 (commit blocked, then unblocked via `git -C`) |
| 2026-05-12 | process/spec-verification | **Verifying that a CF's CODE is correct is not the same as verifying that the CF is ACTUALLY CALLED in production.** PR #58 spec assumed `registerUser` callable runs on every fresh signup — true on paper, false in practice. The callable has been throwing `unauthenticated` on every call (Firebase v1-vs-v2 callable signature bug from the v7 upgrade) and `signup-service.js:109-111` silently swallows the error and falls back to direct RTDB writes. I inlined the day-zero seed inside `registerUser`, code-reviewed it 4×, deployed it to prod, and only discovered it never ran when the operator did the preview e2e test and reported "no workflow activated". **Heuristic: before designing a hook point inside an existing CF, verify the CF actually executes in production — either by reading recent `firebase functions:log --only <fn>` output for the success path, or by deploying a small instrumentation commit and checking logs. Static code review can't see "the CF was never reached".** Sister to 2026-05-12 "preview-channel testing only validates client/hosting" — applies even when the hook is server-side. | Session: PR #58 (operator caught on preview) |
| 2026-05-12 | process/error-swallow | **`try/catch` around a callable in client code can hide a long-running prod bug for months.** `signup-service.js:109-111` wraps `registerUserFunction(...)` in try/catch with `console.warn` then sets `registrationSuccessful = false` and runs the fallback path. The fallback works end-to-end (creates user, sub, location), so signup completes successfully. The callable failure goes to `console.warn` only — no telemetry, no error reporter, no user-visible signal. **Heuristic: when wrapping a server call in client-side try/catch with a fallback, the catch handler should at minimum bump an error counter, emit telemetry, or log to a queryable surface — NOT only `console.warn` which dies with the user's browser session.** Sister to the 2026-05-11 "structural vs content" pattern — the swallowing structure looked right; what was missing was the visibility. | Session: PR #58 (preview test diagnosed via CF logs) |

| 2026-05-12 | rtdb/security-rules | **`.validate` rules under a node with `.write: false` are unreachable; Admin SDK writes (via Cloud Functions) ALWAYS bypass `.validate`, regardless of the parent `.write`.** PR #51 Task 8 originally added a `.validate` on `ross/templates/$id/tier` and `ross/auditLog/templateActivationDenials/$pushId`.
