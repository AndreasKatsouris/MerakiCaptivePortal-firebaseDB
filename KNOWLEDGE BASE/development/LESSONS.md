| 2026-05-19 | process/spec-vs-write-path | **A schema doc / spec describes the shape an artifact is SUPPOSED to have. Verify the actual `set()` / `update()` / `ref().child().set()` call in the corresponding CF before treating any field as guaranteed.** PR #72: spec doc said runs have `status: 'in_progress' \| 'completed'` (sourced from KB ROSS.md). Real `rossCreateRun` (functions/ross.js:1378-1389) writes NO `status` field — lifecycle is encoded via `completedAt === null` (in-progress) vs non-null (completed). My helper checked `latestRun.status === 'in_progress'` — always undefined for real data — and the in-progress / overdue branches never fired despite the data being there. Operator caught it on preview after the date-format fix; second consecutive root-cause-fix on the same PR. Same family as PR #28's `t.id` (server uses `templateId`) and the validated 2026-05-01 SELF_OPT — **second occurrence of the same pattern in 18 days.** The validated pattern says "verify field names against the actual server seed / CF write path"; this strengthens it to "ALSO verify lifecycle markers and absent-field assumptions, not just field names". **Heuristic when designing against an existing data model:** for each field your code reads, grep `<field-name>` in the write-side CF and confirm (a) it's actually written, (b) the type matches, (c) the lifecycle marker (status, completedAt, deletedAt) is where you expect. KB docs lie or lag — `set()`/`update()` calls are ground truth. | Session: PR #72 (operator preview, second defect cycle) |
| 2026-05-19 | rtdb/schema | **A single RTDB field can have multiple storage shapes in the wild (number vs ISO string) — defensive normalisation at read time is mandatory.** PR #72: `nextDueDate` on `ross/workflows/.../locations/.../nextDueDate` is stored as an epoch-ms NUMBER per functions/ross.js:703 (initial write), :1150/1282/1502 (numeric arithmetic at multiple read sites). But the KB doc + the spec said `'ISO date string'`. My helper compared as string (`loc.nextDueDate < today`) which silently returned false for number-vs-string comparisons in JavaScript — all bucket branches missed, two active workflows ended up in NO bucket. Operator-edited values via Firebase Console may land as strings, so production is mixed. **Heuristic:** when reading any "date" or "timestamp" field from RTDB, write a defensive normaliser that accepts BOTH number and string and returns a canonical shape. Apply to anything that could plausibly have been written by different code paths over time (legacy data). Same heuristic applied to PR #67's catch-block re-throw: defensive at the boundary. | Session: PR #72 (first preview defect cycle) |

Sister to the 2026-05-15 "scope-discipline inverse" pattern: same family of "the lesson was loaded but the application branch didn't fire". **Heuristic for the doc-writing workflow:** before committing any KB doc that asserts a Firebase / runtime / pattern behaviour, `grep` the LESSONS file for keywords from the assertion and re-read any matching entry. | 2026-05-19 | process/deploy-sequencing | **Tightening RTDB security rules from `.write: true` to anything stricter creates a "no-data-capture" window between rule deploy and client redeploy — the OLD client still tries the now-blocked write, fails silently into its existing try/catch, and the user-facing flow (Meraki redirect, in this case) succeeds anyway. The data loss is invisible without specific telemetry.** PR #73 deploy: rules went out at T+0, hosting redeploy at T+merge (hours later). For every captive-portal submission in that window, guests still got WiFi but no `wifiLogins` record was created. The 2026-05-01 "deploy CFs before merging client" lesson covers the CREATE direction (new CF deployed before client that calls it); this is the COMPLEMENT — rule-tightening makes the old client's writes silently fail, opposite-direction same-class problem. **Heuristic:** if a PR rewrites a write path from direct-client-RTDB to CF-mediated, ALSO plan to merge + hosting-redeploy quickly after the rules deploy (or sequence rules deploy AFTER merge if the brief data loss is unacceptable). Either way, surface the window explicitly in the deploy plan AND in the user notification. Sister to the 2026-05-01 deploy-sequencing pattern. | Session: PR #73 deploy |
| 2026-05-19 | rtdb/limits | **RTDB's WRITE_TOO_BIG limit (~16 MB) applies per write operation, not per client SDK** — switching from client to Admin SDK does not help. Both `client.remove(ref)` and `admin.database().ref(path).remove()` will fail identically once the node crosses the threshold. The Firebase CLI's `firebase database:remove <path>` succeeds because it chunks recursively under the hood. **Fix pattern for in-app "clear node" admin actions:** chunked delete CF using `orderByKey().limitToFirst(BATCH_SIZE)` + multi-path null `update()` per batch. Hard-cap batches per invocation and return `{ done }` so client can loop across calls for very large nodes. **Operator unblock pattern:** `firebase database:remove <path>` from CLI works immediately while the durable code fix is in flight — don't make the operator wait. Sister to the 2026-05-12 "verify CF actually executes in prod" lesson: same family of "code looks right but the runtime characteristic kills it". | Session: PR #70 (admin "Clear Scanning Data" button) |
| 2026-05-19 | tools/firebase-cli | **`firebase deploy --only functions:X,functions:Y` rejects names that don't exist in the current build** — even if Y exists in deployed prod. Removing an `exports.tempClearData` line from `index.js` and trying `--only functions:tempClearData` errors with `No function matches the filter: default:tempClearData`. The deploy filter checks the BUILD, not what's deployed. **Correct pattern for in-PR function removal:** (a) deploy the new code with `--only functions:<new_or_changed>`, then (b) explicitly `firebase functions:delete <removed_name> --force` for each removed function. A bare `firebase deploy --only functions` would also clean up the removed function, but is slower (touches every function) and riskier. Surfaced when PR #70 tried to combine an update + a deletion in one filtered deploy. | Session: PR #70 deploy |
| 2026-05-19 | process/admin-actions | **Pre-existing in-app admin "clear node" buttons inherit a hidden time-bomb: the underlying node has no retention policy, and the button will fail the day the node crosses 16 MB.** PR #70 root cause: `/scanningData` had been accumulating Meraki captive-portal scans since the project started; nothing ever pruned them; the admin button was the only sink. Heuristic for future admin destructive-action audits: for every "Clear X" / "Delete all Y" admin button, ask (1) "what writes to X?" and (2) "what's the expected steady-state size?" If (1) is unbounded and (2) is "depends how long since last manual clear", you have a latent WRITE_TOO_BIG ticking. Pair every such button with a scheduled prune CF, OR document the operator-must-clear cadence somewhere queryable. Sister to the security-rules audit: same admin surface, same neglected-by-default review category. | Session: PR #70 root-cause analysis |

| 2026-05-15 | tools/git-rebase | **Squash-merged sibling reflect commits cause predictable doc conflicts on rebase — they always touch the same 4 feedback files + project-status.json.** PR #65's mid-PR reflect commit (876cf206) conflicted with master's PR #64 housekeeping reflect on exactly these 5 files: PROJECT_BACKLOG / LESSONS / SCORECARD / SELF_OPTIMIZATION / project-status.json. | 2026-05-13 | tools/git | **The `block-master-commit` hook resolves `cd <path>` from the shell's starting dir; `cd C:/...worktrees/...` from a master-shell triggers it.** When the harness opens a Bash subprocess in the master worktree and you run `cd .worktrees/<name> && git commit ...`, the hook sees a master shell + a destructive git op and blocks even though the resolved cwd is a feature branch. The same applies to `git -C <path> add`, `git -C <path> push`, etc. Worth promoting into the worktree workflow as the default git invocation form when operating across worktrees from a single Claude Code session. | Session: PR #65 (commit blocked, then unblocked via `git -C`) |
