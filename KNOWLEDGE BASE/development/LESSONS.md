| 2026-06-01 | deploy/firebase-cli | **A `firebase deploy --only functions:X` analyses the WHOLE codebase, so an undeployed `defineSecret` blocks ALL function deploys — and the CLI hangs on exit after a large deploy.** Deploying the billing CFs surfaced that `MERAKI_SHARED_SECRET` + `INITIAL_ADMIN_SETUP_SECRET` (introduced by OWASP #89's `defineSecret` migration) were **never provisioned** → non-interactive deploy errors `no value for the secret`. | 2026-05-31 | tools/git-commit | **Backticks inside a double-quoted `git commit -m "..."` are shell command-substitution — they execute, mangling the message.** Phase 7 ledger build: a commit body with `` keyset `before` pagination `` ran `before` as a command (`bash: before: command not found`); the commit still landed but with the backtick content stripped/evaluated. **Fix: for any commit message containing backticks/`$`/parens, use a single-quoted heredoc — `git commit -m "$(cat <<'EOF' ... | 2026-05-31 | agent/no-fabrication | **An LLM agent with write tools must be forbidden — server-side, not just by prompt — from auto-submitting any value it cannot actually obtain or verify; otherwise it will fabricate records.** askRoss Agent spec review (#110): `submitResponse` was tiered `auto` as "drudgery Ross absorbs". But ROSS compliance templates (Health & Safety Audit, Compliance Sweep) carry `inputType: temperature`/`yes_no` regulatory measurements — the agent has no sensors, so auto-submitting a value would **fabricate a food-safety compliance record** (regulatory/liability risk, and a direct violation of "amplify, not replace"). The 422 `requiredNote` net stops an *incomplete* submission but NOT a *fabricated primary value*. Fix (spec §3.1): `submitResponse` auto only for non-measurement types (`text`/`checkbox`/`signature`, `requiredNote` false); measurement/attestation types escalate to the owner; **enforced in the adapter server-side + as a hard eval assertion**, not left to model behaviour. **Heuristic: for every agent write-tool, ask "can the agent legitimately KNOW this value?" — if it asserts a real-world measurement/verification the agent didn't perform, the tool must refuse or escalate, in code.** Sister to auth-vs-authz: defence that doesn't depend on the model (or the user) behaving. | Session: askRoss Agent spec (PR #110 review) |

| 2026-05-31 | rtdb/rules-cascade | **RTDB read permission CASCADES from a parent, and a query is authorised at the QUERIED node — rules cannot filter query results.** Two consequences that drove the whole H-2 design (PR #96): (1) a root `".read": "auth != null"` makes every properly-scoped per-child rule below it DEAD — the node is fully readable regardless. So "tightening" means removing/admin-scoping the ROOT, not just fixing children. (2) You cannot express "let a user query `guests` where `locationId` is theirs" — `query(ref('guests'), orderByChild('locationId'), equalTo(x))` is auth'd at the `guests` root and rules don't post-filter. So a node read via a root-level location-*query* by non-admins CANNOT have its root locked without changing the client read path (→ that's why guests/queue became "Bucket C", deferred for a restructure). Nodes read by **direct keyed path** (`users/$uid`, `salesDataIndex/byLocation/$locId`) CAN be locked + given a scoped child rule with no client change. **Heuristic before touching any root `.read`: map every consumer and classify each as direct-path (lockable now) vs root-query (needs refactor first). Use the `userLocations/$uid/$locId` cross-ref pattern (already in `stockFlagAudit`/`stockItemFlags`) for location scoping — and include the `locations/$locId/ownerId === auth.uid` fallback (automated review caught its omission as a must-fix: users with unpopulated userLocations would lose their own data).** Sister to auth-vs-authz + the field-verify SELF_OPT — verify ground truth before changing behaviour, here applied to rule cascade + query semantics. | Session: PR #96 (H-2 cross-tenant design + review must-fix) |
| 2026-05-30 | security/auth-vs-authz | **In a multi-tenant app, "require a valid token" is NOT "require authorization" — operators/customers hold real Firebase Auth accounts, so a token-only gate still exposes admin-only data to every signed-up non-admin.** PR #91: I first secured `getGoogleConfig` (returns a billable Google Places key) with `verifyIdToken` only. | 2026-05-19 | process/deploy-sequencing | **Tightening RTDB security rules from `.write: true` to anything stricter creates a "no-data-capture" window between rule deploy and client redeploy — the OLD client still tries the now-blocked write, fails silently into its existing try/catch, and the user-facing flow (Meraki redirect, in this case) succeeds anyway. | 2026-05-19 | rtdb/limits | **RTDB's WRITE_TOO_BIG limit (~16 MB) applies per write operation, not per client SDK** — switching from client to Admin SDK does not help. Both `client.remove(ref)` and `admin.database().ref(path).remove()` will fail identically once the node crosses the threshold. The Firebase CLI's `firebase database:remove <path>` succeeds because it chunks recursively under the hood. **Fix pattern for in-app "clear node" admin actions:** chunked delete CF using `orderByKey().limitToFirst(BATCH_SIZE)` + multi-path null `update()` per batch. Hard-cap batches per invocation and return `{ done }` so client can loop across calls for very large nodes. **Operator unblock pattern:** `firebase database:remove <path>` from CLI works immediately while the durable code fix is in flight — don't make the operator wait. Sister to the 2026-05-12 "verify CF actually executes in prod" lesson: same family of "code looks right but the runtime characteristic kills it". | Session: PR #70 (admin "Clear Scanning Data" button) |
