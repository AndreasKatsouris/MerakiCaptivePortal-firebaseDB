ROSS Run Execution UI — Design Spec

Date: 2026-05-13 · Sprint: Hi-Fi v2 with ROSS as central post-login surface — Phase 6 PR 3 · Predecessor: PR #58 (day-zero auto-activation) · Successor: Concierge home active-run surfacing (next sprint task)

Context

The ROSS Runs server surface (rossCreateRun, rossSubmitResponse, rossGetRun, rossGetRunHistory) has been shipped for some time. It supports the full 10-inputType matrix, auto-flagging of out-of-range values, server-side requiredNote enforcement via HTTP 422, idempotent run creation, and auto-completion when all required tasks have responses.

It has zero UI callers. The v1 admin module (public/js/modules/ross/index.js) uses rossCompleteTask for checkbox-style task completion — no runs, no responses, no inputType awareness. As a result, a fresh operator who lands on ROSS with a seeded Daily Opening Checklist (post PR #58) has nothing to actually do with the seeded workflow.

This spec defines the operator-facing Run execution UI: the v2 surface where a manager taps "Start run" on a workflow card, works through its tasks honouring all 10 inputTypes, and lands on a completion banner showing on-time + flagged-count telemetry.

Strategic decisions (locked during brainstorm 2026-05-13)

QDecision
Where does the Run UI mount structurally? New tab on the existing concierge shell: /ross.html?tab=run&workflowId=W&locationId=L. Reuses the existing URLSearchParams + popstate tab routing in RossHome.vue. Deep-linkable, browser-back works, identity-aware sidebar/footer stay visible. Not a sidebar destination — only linkable-to from Playbook cards (and, next sprint, concierge home cards).
How do tasks lay out? Single scrollable list of task cards top-to-bottom, with a sticky bottom summary bar. Operator scrolls and jumps freely (matches real-world spatial behaviour — temp check first because the fridge is right there, then back to "lights on"). No wizard, no accordion, no two-pane.
When does a response persist? Auto-save on commit, per task. Each input commits on its native semantics (toggle / blur / select / tap / star tap). Small "Saving…/Saved" indicator per card. No per-task Save button. Pairs naturally with the server's idempotent run + one-response-at-a-time API.
How is the requiredNote / out-of-range flow surfaced? Client pre-flight + server safety net. On commit of a number/temperature input, the client reads inputConfig.min/max/requiredNote. If out of range AND requiredNote: true, the note textarea is revealed inline before the network round-trip. Value + note submit together. Server still enforces 422 if anyone bypasses the client; the same UI code path handles both.
What does the operator see at run completion? Inline completion banner; stay on the run page. Sticky bar flips to a green completion banner showing onTime + flaggedCount chips and two ghost buttons: Back to Playbook and View in Activity. Run cards become read-only (inputs disabled, checkmarks shown) so the operator can spot-review their submissions. No surprise redirects.
How are photo and signature inputs handled? Render as "Coming soon" placeholder cards with a single Mark N/A ghost button that submits the sentinel string 'n/a' as the response value. This prevents these task types from blocking required-task auto-completion. The server's validateResponseValue has no case for photo/signature, so any string passes type-validation. Upload pipeline + signature pad are Phase 2; out of scope here.

Data layer

run-service.js

Thin wrappers over the three Run CFs. Follows the same id-token / fetch / error-shape pattern as playbook-service.js and people-service.js. No new patterns introduced.

createRun({ workflowId, locationId })
    → POST rossCreateRun
    → returns { runId, status, responses }   // idempotent: returns existing in-progress run if one exists

submitResponse({ workflowId, locationId, runId, taskId, value, note? })
    → POST rossSubmitResponse
    → returns 200 with updated run, OR
    → returns 422 with { error, requiredNote: true }
    → 422 is NOT thrown — service surfaces { status: 422, requiredNote: true } so the store can react

getRun({ workflowId, locationId })
    → POST rossGetRun
    → returns { currentRun, previousResponses }   // used on mount to resume

run-store.js (Pinia)

state:
  currentRun:  null | { runId, status, startedAt, completedAt?, onTime?, flaggedCount? }
  workflow:    null | { workflowId, name, tasks[] }   // task[] carries inputType + inputConfig + required
  responses:   { [taskId]: { value, note?, submittedAt, flagged } }
  saveStatus:  { [taskId]: 'idle' | 'saving' | 'saved' | 'error' }
  errors:      { [taskId]: string | null }            // server message verbatim per People-tab pattern
  loading:     boolean
  loadError:   string | null

actions:
  initRun(workflowId, locationId)
      // 1. getRun  → hydrate responses if in-progress
      // 2. createRun → idempotent; returns same run if mid-flight
      // 3. load workflow definition for the per-task inputType/inputConfig/required fields
  commitResponse(taskId, value, note?)
      // optimistic: saveStatus='saving' → service.submitResponse → handle 200/422/other
  dismissError(taskId)
  reset()

Component responsibilities

ComponentOwnsDoes not own
RossRun.vue Mount + route param read; calls store.initRun; renders page-level loading / error / empty states; renders task list + summary bar / completion banner. Per-task state, input rendering, save mechanics.
RossRunTaskCard.vue One task's frame — title, required pill, description, hosts the input, hosts the inline note textarea when revealed, shows saveStatus indicator, shows per-card error banner with dismiss button. Server calls, input semantics, range validation.
RossRunTaskInput.vue The input itself. Switches on inputType internally. Emits commit(value) on per-type commit semantics. Emits preflightNote(reason) when client-side range check trips on number / temperature with requiredNote: true. Saving (delegates up), note text capture (parent owns), persistence.
RossRunSummaryBar.vue Sticky bottom bar: progress count (required_done / required_total), Finish-and-leave button. Server auto-completes; Finish is a navigation affordance, not a submit. Run state, task data.
RossRunCompletionBanner.vue Mounts when currentRun.status === 'completed'. Renders success copy + onTime + flaggedCount chips + two ghost buttons (Back to Playbook, View in Activity). Data via props. Routing decisions (parent decides which URL the buttons go to).

Key flows

Start / resume

  1. RossHome.vue sees ?tab=run&workflowId=W&locationId=L, mounts RossRun.vue with those as props.
  2. RossRun.onMounted calls store.initRun(W, L):
    1. getRun first — if an in-progress run exists, hydrate currentRun + responses from the server response.
    2. createRun next — server returns the same in-progress run (idempotent) or creates a fresh one.
    3. Load workflow definition so we have task[] with inputType, inputConfig, required.
  3. Render. Tasks with existing responses show pre-filled, with a checkmark and Saved indicator.

Save-on-commit (one task)

User commits value in RossRunTaskInput
    ↓ emit commit(value)
RossRunTaskCard runs pre-flight check for number/temperature:
    if out-of-range AND inputConfig.requiredNote === true:
        reveal note textarea, focus it, do NOT save yet
        wait for note input + blur
    ↓ emit save({ value, note? })
RossRun → store.commitResponse(taskId, value, note?)
    saveStatus[taskId] = 'saving'
    ↓ service.submitResponse(...)
        ├─ 200 → responses[taskId] = updated;  saveStatus='saved'  (fade to 'idle' after 1.5s)
        ├─ 422 → re-emit preflightNote to card; saveStatus='idle'  (note flow is expected, not an error)
        └─ other error → errors[taskId] = message; saveStatus='error'

If the 200 response indicates the run flipped to completed, store updates currentRun.status and RossRun.vue reactively renders RossRunCompletionBanner in place of the summary bar; cards transition to read-only.

Photo / signature placeholder

Card renders a disabled visual control (icon + label "Photo capture" or "Signature") with a Coming soon tag and a single ghost button Mark N/A. Tapping the button submits value: 'n/a' via the standard commit flow. The server's validateResponseValue has no case for photo/signature, so any string passes type-validation. The task counts as responded and does not block run auto-completion.

Template authoring note: If a template author sets a photo/signature task as required: true, the operator can still complete via Mark N/A. This is intentional for v1 — the alternative is blocking completion until the upload pipeline ships, which would be worse. Template authors can mitigate by marking these tasks required: false until Phase 2.

Leave and return

Operator navigates away mid-run (back to Playbook, closes tab, signs out and back in tomorrow morning). store.reset() fires on unmount. Returning to the same URL re-runs initRun; rossGetRun returns the same in-progress run; responses hydrate. Fully idempotent.

Error handling

FailureUX
Per-task save failure (non-422) Error banner inside the affected card with server message verbatim + dismiss button. Same pattern as People tab. Retry happens implicitly when operator re-commits.
initRun failure Page-level error banner with Retry button. Back to Playbook link always present so user is never stranded.
Network offline mid-run saveStatus stays 'saving' until the request resolves or fails. No client-side queue. Operator re-commits when reconnected. Documented limitation; offline-queue is a separate enhancement.
Stale token / 401 Service catches 401, store surfaces "Session expired — sign in again" at page level. Mirrors existing v2 services.

Testing strategy

Unit (vitest)

  • run-store.test.js — initRun happy path; initRun resumes mid-run (responses hydrate); commitResponse 200 / 422 / other-error branches; commitResponse flips currentRun.status to completed when server response indicates auto-completion.
  • RossRunTaskInput.preflight.test.js — pre-flight reveals note field for out-of-range temperature with requiredNote: true; does NOT reveal for in-range; does NOT reveal when requiredNote: false; does NOT reveal for non-numeric input types.

Coverage target: 80%+ on run-store.js and the pre-flight logic in RossRunTaskInput.vue. Light coverage on presentational components.

Manual smoke (before claiming done)

  1. Start a new run of the seeded Daily Opening Checklist; complete all tasks; verify completion banner shows onTime + flaggedCount chips.
  2. Start a run, exit mid-way, reopen the URL — responses hydrate, focus lands on the first unanswered task.
  3. Enter 12°C on a fridge-temp task with min: 0, max: 5, requiredNote: true; verify note field reveals client-side with no round-trip; save with note; verify saved + flagged.
  4. Enter 12°C then clear the note textarea — verify save is blocked / value is not persisted.
  5. Mobile breakpoint (≤640px) — full smoke on a phone or DevTools mobile.
  6. Photo / signature Mark N/A completes the task and counts toward auto-completion.

Preview channel

Deploy from worktree before requesting operator preview validation. CFs need no deploy (no server changes). Per LESSONS pattern: preview validates client/hosting; since this PR has no CF changes, preview is sufficient verification.

Out of scope (explicit non-goals)

  • Concierge home active-run surfacing — separate sprint task. This PR only adds the Run UI; the home cards remain scripted until that follow-up.
  • Photo upload pipeline — Phase 2. Cloud Storage bucket wiring, image compression, thumbnails, etc.
  • Signature pad — Phase 2. Canvas-based signature capture + storage.
  • Offline queue — saves that fire while disconnected are not retried automatically. Documented limitation.
  • Operator-initiated "Cancel run" — the server has no rossCancelRun. Runs that the operator abandons remain in_progress indefinitely until completed.
  • Run history / re-open completed runs — that's the Activity tab's job. Completion banner links there.
  • Mobile bottom-nav addition — Run is link-only, not a destination users browse to. (The pre-existing "mobile nav missing Playbook/Activity/People" backlog bug is independent and untouched.)

Risks

RiskMitigation
Auto-save on every commit could spam the server if the operator rapidly toggles a checkbox. Acceptable for v1 — server is idempotent on the same response; checkbox toggling is operator behaviour, not a hot loop. If load becomes an issue, add a 250ms debounce on the commit emit.
Client pre-flight diverges from server validation logic if either changes. Server is the safety net — its 422 path is exercised by the same UI code as pre-flight reveal. If the client check drifts, the server path still surfaces correctly. Unit test asserts both paths share UX.
Hi-Fi token names: same family of bugs as PR #55 (used non-existent --hf-fg, --hf-surface, etc.). Verify all --hf-* tokens against public/css/hifi-tokens.css before authoring any scoped style block. Listed in LESSONS 2026-05-12.
Worktree Read paths: Edit silently rejects files Read from master path while in a worktree. Read every file from the worktree path. Asset-hash delta after npm run build is the verification signal, not just build-green. Listed in LESSONS 2026-05-11.
Double-click "Start run" race could theoretically create duplicate runs. rossCreateRun is server-idempotent — returns the existing in-progress run. No client-side guard needed.
