Critical Severity Findings

C-1 · A01 Broken Access Control
Operator precedence bug bypasses admin check on 12 Cloud Functions
A01 functions/index.js:726,1521+ ▼
What it is

The expression !decodedToken.admin === true evaluates as (!decodedToken.admin) === true. When decodedToken.admin is true (i.e., an actual admin), this returns false — meaning the guard never blocks anyone. Every authenticated user passes the check.

Affected functions: setAdminClaim, markVoucherRedeemed, getVoucherDetails, getVoucherPoolAvailability, updateVoucherPoolStats, bulkQueueOperations, sendQueueNotification, sendManualQueueAdditionNotification, cleanupOldQueues, getQueueAnalytics, getRealtimeQueueMetrics, getQueuePerformanceStats.

Evidence
// functions/index.js:726 — allows anyone to promote users to admin
if (!callerToken.admin === true) {   // BUG: (!true) === true → false → never blocks

// functions/index.js:1521 — allows any auth user to redeem vouchers
if (!decodedToken.admin === true || !isAdminInDb) {
Fix
Replace all 12 occurrences: !decodedToken.admin === true → decodedToken.admin !== true
Also update compound checks: decodedToken.admin !== true || !isAdminInDb
C-2 · A02 Cryptographic Failures / A05 Security Misconfiguration
Hardcoded production secrets in committed source code
A02A05 functions/index.js:443,1250 ▼
What it is

Two secrets are hardcoded in version-controlled source code. (1) The Meraki webhook shared secret — anyone who knows this can POST arbitrary scanning data. (2) The admin bootstrap secret — the setupInitialAdmin endpoint promotes any email to platform admin with no secondary guard.

Evidence
// functions/index.js:443
const sharedSecret = 'Giulietta!16'; // Replace with your shared secret

// functions/index.js:1250
const SETUP_SECRET = 'MerakiAdmin2024!'; // You should change this!
Fix
1. Rotate both secrets immediately.
2. Move to Firebase Secrets Manager: firebase functions:secrets:set MERAKI_SHARED_SECRET
3. Reference via process.env.MERAKI_SHARED_SECRET in the function.
4. Gate setupInitialAdmin with if (process.env.INITIAL_SETUP_ENABLED !== 'true') return res.status(404).end() — disabled by default in production.
C-3 · A02 Cryptographic Failures
Google Places API key exposed to unauthenticated public via getGoogleConfig
A02 functions/index.js:683 ▼
What it is

The getGoogleConfig Cloud Function returns the Google Places API key and Place ID with zero authentication. Any person on the internet can call this endpoint and receive a billable API key.

Evidence
// functions/index.js:683
exports.getGoogleConfig = onRequest(async (req, res) => {
    // No auth check before this:
    res.json({
        apiKey: functions.config.google.places_api_key,
        placeId: functions.config.google.place_id
    });
});
Fix
Option A (preferred): Embed the key at build time (Vite env var) and remove the endpoint entirely. Apply HTTP referrer restriction to the key in Google Cloud Console.
Option B: Add a valid Firebase ID token check: verify req.headers.authorization via admin.auth().verifyIdToken() before responding.

Change both to: ".write": "auth != null && auth.token.admin === true"
C-5 · A10 Server-Side Request Forgery
ocrReceipt passes user-supplied imageUrl directly to Google Vision API (SSRF)
A10 functions/index.js:3542 ▼
What it is

The ocrReceipt endpoint accepts imageUrl from the request body and passes it directly to detectReceiptText(imageUrl) → client.textDetection(imageUrl). An attacker with admin credentials can direct the Cloud Vision API to fetch arbitrary URLs, including internal GCP metadata endpoints. The same URL is also used in receiptProcessor.js with a bypass-able .includes('twilio.com') check (a hostname like evil.twilio.com.attacker.com passes).

Evidence
// functions/index.js:3515
const { imageUrl, imageData } = req.body;
...
} else {
    [result] = await detectReceiptText(imageUrl);  // no allowlist — SSRF
}

// functions/receiptProcessor.js:310
if (imageUrl.includes('twilio.com')) {  // bypassable hostname check
Fix
Before calling Vision API or any fetch, validate against an explicit allowlist:
const ALLOWED_PREFIXES = [
  'https://firebasestorage.googleapis.com/',
  'https://storage.googleapis.com/',
];
const isAllowed = ALLOWED_PREFIXES.some(p => imageUrl.startsWith(p));
if (!isAllowed) return res.status(400).json({ error: 'Invalid image URL' });
Also fix the receiptProcessor.js check to use new URL(imageUrl).hostname.endsWith('.twilio.com').

H-1 · A01 Broken Access Control
5 queue management Cloud Functions have no authentication
A01 functions/index.js:1692–1909 ▼
What it is

Five public endpoints — addGuestToQueue, removeGuestFromQueue, updateQueueEntryStatus, getQueueStatus, processQueueMessage — pass req.body directly to queue logic without any token verification. Any unauthenticated caller can add/remove guests or mark entries as served at any restaurant.

Evidence
exports.addGuestToQueue = functions.https.onRequest(async (req, res) => {
    return cors(req, res, async () => {
        // No auth check — req.body passed directly
        const result = await addGuestToQueue(req.body);
    });
});
Fix
Add verifyIdToken (or Firebase anonymous auth check) at the top of all five handlers before executing queue logic. Admin operations additionally require the admin claim.
H-2 · A01 Broken Access Control
guests node (+ 8 others) readable by any authenticated user — cross-tenant PII leak
A01 database.rules.json:13 ▼
What it is

The root guests node has .read: "auth != null". Any authenticated user — including a patron who authenticated via WiFi login — can read the entire guest database across all tenants: names, phone numbers, emails, marketing consent flags. Same issue affects users, subscriptions, receipts, salesData, salesDataIndex, forecasts, forecastIndex, and queue.

Evidence
// database.rules.json:13
"guests": {
  ".read": "auth != null",  // ALL authenticated users — no tenant scoping
  ...
}
Fix
Scope root reads to "auth.token.admin === true". Grant per-record reads to the owning tenant only (via location claim or userLocations cross-reference). Apply consistently to all 9 affected nodes.

H-4 · A02 Cryptographic Failures
Math.random() used for temporary password generation (non-CSPRNG)
A02 functions/index.js:1157 ▼
What it is

Temporary passwords are generated using Math.random() in two places. Math.random() is not a cryptographically secure PRNG — its output can be predicted if the seed state is known.

Evidence
// functions/index.js:1157
tempPassword += chars.charAt(Math.floor(Math.random() * chars.length));

// enhanced-user-subscription-manager.js:886
password += chars.charAt(Math.floor(Math.random() * chars.length));
Fix
In Cloud Functions (Node.js): const { randomInt } = require('crypto'); randomInt(0, chars.length)
In client-side JS: window.crypto.getRandomValues(new Uint32Array(1))[0] % chars.length
H-5 · A05 Security Misconfiguration
CORS allows any origin on all ROSS Cloud Function endpoints
A05 functions/ross.js:15 ▼
What it is

All ROSS Cloud Functions use cors({ origin: true }), which reflects any origin. Any website can make cross-origin requests to ROSS endpoints (workflow templates, staff data, audit logs, run records) while using the victim admin's browser credentials. functions/index.js already has an isAllowedOrigin allowlist — ross.js doesn't use it.

Evidence
// functions/ross.js:15
const cors = require('cors')({ origin: true });  // reflects ANY origin
Fix
Import and reuse the isAllowedOrigin function from index.js:
const cors = require('cors')({ origin: isAllowedOrigin });

What it is

setAdminClaim is an onRequest function that does not apply the cors middleware, unlike other state-changing endpoints. Combined with C-1's broken admin check, this endpoint can be called from any origin without CORS preflight restrictions.

H-7 · A09 Security Logging
Full request headers (including Authorization JWT) logged to Cloud Logging
A09 functions/index.js:697,770,858 ▼
What it is

Three functions log headers: req.headers in their entirety. The req.headers object includes the Authorization: Bearer <JWT> header. Firebase ID tokens logged in cleartext can be extracted and used to impersonate users during the token's up-to-1-hour validity window.

Evidence
// functions/index.js:858 (createUserAccount)
console.log('[createUserAccount] Received request:', {
    method: req.method,
    headers: req.headers   // ← includes Authorization: Bearer <JWT>
});
// Same pattern at lines 697 (setAdminClaim) and 770 (setupInitialAdmin)
Fix
Never log req.headers. Log only req.method, req.headers['content-type'], and the caller's UID after token verification.
H-8 · A02 Cryptographic Failures
Firebase Storage /receipts/ path has public unauthenticated write access
A02 storage.rules:17 ▼
What it is

The Storage rule for /receipts/{receiptFile} has allow write: if true — unconditional public write. Any unauthenticated actor can upload arbitrary files. Cloud Functions writing to Storage use the Admin SDK which bypasses rules entirely, so this if true is unnecessary and dangerous.

Evidence
// storage.rules:15
match /receipts/{receiptFile} {
  allow read: if request.auth != null;
  allow write: if true;  // ← unauthenticated public write
}
Fix
Change to allow write: if false; — Cloud Functions use the Admin SDK and are unaffected.

M-1 · A04 Insecure Design
Location creation allows arbitrary ownerId — no server-side ownership validation
A04database.rules.json:43▼
What it is

The !data.exists() creation path on $locationId permits any authenticated user to create a location record with ownerId set to any UID. No .validate rule enforces newData.child('ownerId').val() === auth.uid.

Fix
Add: ".validate": "newData.child('ownerId').val() === auth.uid" on the $locationId node, or gate all location creation through a Cloud Function that stamps ownerId server-side.
M-2 · A05 Security Misconfiguration
Missing security headers — CSP, X-Frame-Options, HSTS, X-Content-Type-Options
A05firebase.json▼
What it is

Firebase Hosting only sets Content-Type and Cache-Control. The admin dashboard is frameable from any domain (clickjacking), MIME sniffing is unrestricted, and HTTPS is not enforced via HSTS.

Fix
Add a headers block in firebase.json for "source": "**":
X-Frame-Options: DENY · X-Content-Type-Options: nosniff · Referrer-Policy: strict-origin-when-cross-origin · Strict-Transport-Security: max-age=31536000; includeSubDomains · Content-Security-Policy (scoped to known CDNs).
M-3 · A08 Software and Data Integrity
No Subresource Integrity (SRI) on CDN-loaded scripts across all HTML pages
A08All HTML pages▼
What it is

Every HTML page loads Bootstrap, Vue 3, SweetAlert2, Font Awesome, Chart.js, Lodash, and PapaParse from public CDNs without integrity= attributes. A CDN compromise or BGP hijack would result in arbitrary JS execution on every page including the admin dashboard.

Fix
Generate SRI hashes: openssl dgst -sha384 -binary <file> | openssl base64 -A
Add integrity="sha384-..." and crossorigin="anonymous" to every external <script> and <link>. Pin exact versions (e.g., sweetalert2@11.14.5 not @11).
M-4 · A03 Injection
googleReviews.js renders unescaped review data directly into innerHTML (stored XSS)
A03public/js/googleReviews.js:231▼
What it is

The tableBody.innerHTML assignment interpolates review.reviewerName, review.text, review.response, review.authorUrl directly without calling escapeHtml(). If any review from RTDB or Google Places API contains <script> tags or event handler attributes, it will execute in the admin's browser.

Evidence
// public/js/googleReviews.js:231
tableBody.innerHTML = this.state.reviews.map(review => `
    <div class="fw-bold">${review.reviewerName}</div>  <!-- unescaped -->
    <p class="mb-1">${review.text}</p>                 <!-- unescaped -->
    <p class="mb-0">${review.response}</p>              <!-- unescaped -->
    <a href="${review.authorUrl}">...</a>               <!-- unescaped URL -->
`);
Fix
Wrap every interpolated value with escapeHtml() (already defined in the codebase). For URL fields, additionally validate they start with https:// before use in href.
